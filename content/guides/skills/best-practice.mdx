✅ Good:

<FileBlock filename="SKILL.md — 구체적인 지침 예시">

## 오류 처리 포함

MCP 연동이나 외부 API를 쓰는 스킬은 환경마다 동작이 달라요. Claude는 오류 메시지를 보고 스스로 판단하려 하지만, 도구 특유의 에러 코드나 재연결 절차는 알 수 없어요. 자주 발생하는 오류와 해결 방법을 스킬에 직접 써두면, Claude가 막히는 순간에 사용자에게 되묻지 않고 스스로 복구해요.

<FileBlock filename="SKILL.md — 오류 처리 섹션">
```markdown
## 일반적인 문제 (Common Issues)

### MCP 연결 실패 (MCP Connection Failed)

"Connection refused" 오류가 발생하는 경우:

1. MCP 서버가 실행 중인지 확인:
   Settings > Extensions에서 확인
2. API 키가 유효한지 확인
3. 재연결 시도:
   Settings > Extensions > [사용 중인 서비스] > Reconnect
```
</FileBlock>

## Claude가 이미 아는 것은 쓰지 마세요

Claude Code는 코드베이스를 직접 읽고, Claude 자체도 코딩에 대한 판단 기준과 선호를 이미 가지고 있어요. "코드를 명확하게 작성하세요", "에러를 적절히 처리하세요" 같은 지침은 스킬이 없어도 Claude가 알아서 하는 것들이에요.

지식 중심의 스킬이라면, **Claude의 기본 패턴에서 벗어나게 만드는 정보**에만 집중하세요.

Anthropic 엔지니어들이 고객과 반복 개선하며 만든 `frontend-design` 스킬이 좋은 예예요. 이 스킬은 "좋은 UI를 만드세요"가 아니라, Claude가 혼자 두면 반복적으로 선택하는 패턴들 — Inter 폰트, 보라색 그라디언트, 카드 기반 레이아웃 — 을 명시적으로 피하도록 유도해요.

<Callout type="tip">
스킬을 작성하기 전에 스스로에게 물어보세요: **"이 내용이 없으면 Claude가 다르게 행동하는가?"** 대답이 "아니오"라면, 그 문장은 지워도 돼요.
</Callout>

## Gotchas (주의사항)

어떤 스킬이든 가장 중요한 섹션은 **Gotchas**예요. Claude가 해당 스킬을 사용할 때 자주 실패하는 지점을 모아두는 곳이에요.

스킬을 처음 만들 때는 비어 있어도 괜찮아요. 실제로 써보면서 Claude가 반복적으로 틀리는 지점이 생기면, 그때그때 추가하세요.


Claude가 실패할 때마다 한 줄씩 추가하세요.

## 파일 시스템과 점진적 공개 활용

스킬은 단순한 마크다운 파일이 아니라 **폴더**예요. 전체 파일 구조 자체를 컨텍스트 설계의 일부로 생각해야 해요.

`SKILL.md`에 모든 내용을 몰아넣는 대신, 역할에 맞게 파일을 나눠두면 Claude가 필요한 시점에 필요한 파일만 꺼내 읽어요.

| 폴더 | 용도 |
|------|------|
| `references/` | 상세 API 명세, 스타일 가이드 등 참고 문서 |
| `assets/` | 결과물 템플릿 (마크다운, HTML 등) |
| `scripts/` | 실행 가능한 헬퍼 스크립트 |
| `examples/` | 입출력 예시 |

`SKILL.md` 본문에서 파일의 존재와 용도를 명시해두면, Claude가 언제 꺼내야 하는지 판단할 수 있어요.

<FileBlock filename="SKILL.md">
```markdown
## 추가 자료

- API 요청 형식과 인증 방법은 `references/api.md`를 참고하세요.
- 보고서 출력은 `assets/report-template.md`를 기반으로 작성하세요.
- 데이터 전처리는 `scripts/preprocess.py`를 실행하세요.
```
</FileBlock>

## 스크립트 제공 & 코드 생성 활용

Claude에게 코드를 직접 제공하는 것은 매우 강력해요. 스크립트와 라이브러리를 스킬에 포함하면, Claude는 반복적인 코드를 다시 만들 필요 없이 **무엇을 할지 결정**하는 데 집중할 수 있어요.

<Callout type="tip">
데이터 분석 스킬에서 데이터 조회 함수들을 `scripts/`에 제공하면, Claude가 이를 조합해 고급 분석을 수행해요.
</Callout>

<FileBlock filename="scripts/query.py">
```python
def get_sales(start_date, end_date):
    """날짜 범위의 매출 데이터를 반환"""
    ...

def get_user_cohort(cohort_month):
    """특정 월 가입 사용자의 리텐션 데이터를 반환"""
    ...

def get_funnel(steps: list[str]):
    """단계별 전환율 데이터를 반환"""
    ...
```
</FileBlock>

<FileBlock filename="SKILL.md">
```markdown
# Data Analyst

데이터 분석 요청을 수행합니다.

## 사용 가능한 함수

분석에 필요한 데이터는 `scripts/query.py`의 함수를 실행해 가져오세요.
함수를 조합해 복합 분석을 수행할 수 있습니다.
```
</FileBlock>

Claude는 함수 구현을 알 필요 없이, 어떤 데이터가 필요한지 판단해 함수를 조합해요.

<VibeBlock>
```
지난 분기 신규 가입자 중 3개월 리텐션이 가장 높은 채널이 어딘지 분석해줘
```
</VibeBlock>

<ClaudeBlock>
```
get_user_cohort()로 채널별 신규 가입자를 조회하고,
get_sales()로 해당 기간 매출과 교차 분석해 리텐션 상위 채널을 찾겠습니다.
```
</ClaudeBlock>

## 메모리와 데이터 저장

스킬 내부에 데이터를 저장해 일종의 메모리처럼 활용할 수 있어요.

- 텍스트 로그 파일
- JSON
- SQLite

예를 들어 `standup-post` 스킬이 `standups.log`를 저장하면, 다음 실행 시 이전 결과를 읽고 변화만 반영할 수 있어요.

<Callout type="warning">
스킬 업그레이드 시 스킬 폴더 내 데이터가 삭제될 수 있어요. `${CLAUDE_PLUGIN_DATA}` 같은 안정적인 경로에 저장하는 것이 좋아요.
</Callout>

<FileBlock filename="~/.claude/skills/standup-post/SKILL.md">
```markdown
# Standup Post

오늘의 스탠드업을 작성하고 기록합니다.

## 실행 순서

1. `${CLAUDE_PLUGIN_DATA}/standups.log`를 읽어 직전 스탠드업 내용을 파악합니다.
   (파일이 없으면 첫 실행으로 간주합니다.)
2. 오늘 작업 내용을 바탕으로 스탠드업을 작성합니다.
   - 직전 내용과 비교해 **변화된 부분**을 강조합니다.
3. 작성한 내용을 `${CLAUDE_PLUGIN_DATA}/standups.log`에 날짜와 함께 추가합니다.
```
</FileBlock>

로그 파일은 이렇게 쌓여요.

<FileBlock filename="${CLAUDE_PLUGIN_DATA}/standups.log">
```
[2026-05-12]
어제: API 인증 모듈 설계
오늘: 토큰 갱신 로직 구현 예정
블로커: 없음

[2026-05-13]
어제: 토큰 갱신 로직 구현 완료
오늘: 단위 테스트 작성
블로커: 없음
```
</FileBlock>
