# 스킬 만들기

## 스킬이 저장되는 곳

스킬 폴더는 두 위치 중 하나에 저장해요.

| 저장 위치 | 적용 범위 |
|-----------|-----------|
| `~/.claude/skills/` | 내 컴퓨터의 모든 프로젝트에서 사용 |
| `.claude/skills/` | 현재 프로젝트에서만 사용 |

<Callout type="tip">
`.claude/skills/`에 저장하면 팀 전체가 동일한 스킬을 공유할 수 있어요. Git으로 관리하면 스킬도 코드처럼 버전 관리가 돼요.
</Callout>

---

## 스킬의 구조

스킬은 다음과 같은 구성 요소를 포함하는 **폴더**예요. 하나씩 설명해드릴게요.

| 파일/폴더 | 필수 여부 | 설명 |
|-----------|-----------|------|
| `SKILL.md` | ✅ 필수 | YAML 프론트매터가 포함된 마크다운 형식의 지침 파일 |
| `scripts/` | 선택 | 실행 가능한 코드 (Python, Bash 등) |
| `references/` | 선택 | 필요할 때 로드되는 문서 |
| `assets/` | 선택 | 출력에 사용되는 템플릿, 폰트, 아이콘 |

폴더 구조는 이렇게 생겼어요.

<FileTree root={{
  name: "code-review/",
  children: [
    { name: "SKILL.md", note: "✅ 필수. 지침 본문" },
    { name: "scripts/", note: "선택. 실행 스크립트", children: [
      { name: "check.py" },
    ]},
    { name: "references/", note: "선택. 참고 문서", children: [
      { name: "style-guide.md" },
    ]},
    { name: "assets/", note: "선택. 템플릿·아이콘 등", children: [
      { name: "report.md" },
    ]},
  ]
}} />

---

이제 실제로 스킬을 하나씩 만들어볼게요.

그런데 파일을 바로 만들기 전에 먼저 해야 할 게 있어요.\
**어떤 스킬을 왜 만들지** — 즉 기획과 설계를 먼저 해야 해요.

스킬을 잘 만들려면 "어떤 상황에서 Claude가 이 스킬을 꺼내야 하는가", "어떤 결과가 나오면 성공인가"를 미리 정해두는 게 중요해요. 이걸 건너뛰면 description을 어떻게 써야 할지 막막하고, 만들고 나서도 "뭔가 아닌 것 같은" 스킬이 되기 쉬워요.

## <StepBadge step={1} /> 기획 및 설계

### 유즈케이스부터 시작하기

코드를 작성하기 전에, 해당 스킬이 가능하게 해야 할 **구체적인 유즈케이스 2~3개**를 먼저 정의하세요.

좋은 유즈케이스 정의를 위해 다음 질문을 스스로에게 해보세요.

- 사용자는 무엇을 달성하고 싶은가?
- 이를 위해 어떤 **다단계 워크플로우**가 필요한가?
- 어떤 도구가 필요한가? (내장 도구 또는 MCP)
- 어떤 도메인 지식이나 모범 사례를 포함해야 하는가?

유즈케이스를 이렇게 구체적으로 적어두면 SKILL.md 본문 작성이 훨씬 쉬워져요.

<FileBlock filename="usecase-example.md">
```
유즈케이스: 프로젝트 스프린트 계획
트리거: 사용자가 "스프린트 계획을 도와줘" 또는 "스프린트 작업을 만들어줘"라고 말할 때

단계:
1. Linear에서 현재 프로젝트 상태 가져오기 (MCP를 통해)
2. 팀의 속도(velocity)와 작업 가능 용량(capacity) 분석
3. 작업 우선순위 제안
4. 적절한 라벨과 예상 소요 시간을 포함해 Linear에 작업 생성

결과: 작업이 생성된 상태의 완전한 스프린트 계획 완료
```
</FileBlock>

### 유즈케이스 카테고리

Anthropic이 관찰한 세 가지 주요 유즈케이스 카테고리예요. 스킬을 만들기 전에 어디에 해당하는지 파악하면 설계 방향이 명확해져요.

| 카테고리 | 설명 | 예시 스킬 | 핵심 기법 |
|----------|------|-----------|-----------|
| **문서 및 자산 생성** | 문서·앱·디자인·코드 등 일관되고 높은 품질의 결과물 생성 | `frontend-design` | 스타일 가이드 내장, 템플릿 구조, 품질 체크리스트 |
| **워크플로우 자동화** | 일관된 방법론이 필요한 다단계 프로세스 자동화. MCP 서버 간 협업 포함 | `skill-creator` | 검증 단계 포함, 템플릿 제공, 반복적 개선 루프 |
| **MCP 확장** | MCP 도구 접근을 효과적으로 활용하도록 워크플로우 가이드 제공 | `sentry-code-review` | MCP 호출 순차 실행, 도메인 지식 내장, 오류 처리 |

---

### 성공 기준 정의

스킬이 잘 작동하고 있는지 판단할 기준을 미리 정해두세요. 아래 지표는 엄밀한 임계값보다 **지향점**에 가까운 참고 기준이에요.

**정량적 지표**

| 지표 | 측정 방법 |
|------|-----------|
| 관련 질의의 90%에서 스킬이 트리거됨 | 테스트 질의 10~20개 실행 → 자동 로드 횟수 vs 수동 호출 횟수 비교 |
| 워크플로우를 N번의 툴 호출로 완료 | 스킬 사용·미사용 상태에서 각각 실행 → 툴 호출 횟수·토큰 사용량 비교 |
| 워크플로우당 API 호출 실패 0건 | 테스트 중 MCP 서버 로그 모니터링 → 재시도 비율·에러 코드 추적 |

**정성적 지표**

| 지표 | 평가 방법 |
|------|-----------|
| 사용자가 다음 단계를 다시 물어볼 필요 없음 | 테스트 중 방향 수정·추가 설명이 필요한 횟수 기록, 베타 피드백 수집 |
| 수정 없이 워크플로우가 완료됨 | 동일 요청을 3~5회 실행 → 결과의 구조적 일관성·품질 비교 |
| 세션 간 일관된 결과 제공 | 신규 사용자가 최소한의 안내로 첫 시도에서 작업을 완료할 수 있는지 확인 |

## <StepBadge step={2} /> 스킬 폴더 만들기

스킬은 파일이 아니라 **폴더** 단위예요. 폴더를 하나 만드는 것부터 시작해요.

<ShellBlock>
```bash
# 전체 프로젝트에서 사용 (글로벌)
mkdir ~/.claude/skills/explain-code

# 현재 프로젝트에서만 사용 (프로젝트)
mkdir -p .claude/skills/explain-code
```
</ShellBlock>

이 **폴더명이 곧 스킬명**이에요. Claude가 이 이름으로 스킬을 인식하고, `/explain-code`로 직접 호출할 때도 이 이름을 사용해요.

폴더명은 kebab-case만 사용해요.

| ✅ 허용 | ❌ 금지 |
|---------|---------|
| `explain-code` | `Explain Code` (공백) |
| `code-review` | `code_review` (언더스코어) |
| `notion-setup` | `NotionSetup` (대문자) |

---

## <StepBadge step={3} /> SKILL.md 파일 만들기

폴더 안에 `SKILL.md` 파일을 만들어요.

<ShellBlock>
```bash
touch ~/.claude/skills/explain-code/SKILL.md
# 또는 프로젝트 레벨
touch .claude/skills/explain-code/SKILL.md
```
</ShellBlock>

파일명은 반드시 정확히 `SKILL.md`여야 해요. 대소문자를 구분하기 때문에 다른 형태는 인식되지 않아요.

| ✅ 허용 | ❌ 금지 |
|---------|---------|
| `SKILL.md` | `SKILL.MD` |
| | `skill.md` |
| | `Skill.md` |

## <StepBadge step={4} /> SKILL.md 작성하기

SKILL.md는 두 부분으로 구성돼요.

- **YAML 프론트매터** (`---` 마커 사이) — Claude에게 이 스킬을 언제 사용할지 알려줘요
- **마크다운 본문** — 스킬이 호출됐을 때 Claude가 따르는 지침이에요

`name` 필드는 `/slash-command`가 되고, `description`은 Claude가 자동으로 로드할 시기를 결정하는 데 사용돼요.

`~/.claude/skills/explain-code/SKILL.md`를 이렇게 작성해보세요.

<FileBlock filename="~/.claude/skills/explain-code/SKILL.md">
```markdown
---
name: explain-code
description: 시각적 다이어그램과 비유를 활용해 코드를 설명합니다. 코드가 어떻게
  동작하는지 설명할 때, 코드베이스를 가르칠 때, 또는 사용자가 "이거 어떻게 동작해?"
  라고 물을 때 사용합니다.
---

# Explain Code

## Instructions

### 1단계: 비유로 시작하기

코드를 일상적인 것에 비유하세요.
복잡한 개념이라면 여러 개의 비유를 활용하세요.

### 2단계: 다이어그램 그리기

ASCII 아트를 사용해 흐름, 구조, 관계를 보여주세요.

### 3단계: 코드 단계별 설명

실제로 어떤 일이 일어나는지 순서대로 설명하세요.

### 4단계: 주의할 점 강조하기

흔히 하는 실수나 오해를 짚어주세요.

설명은 대화하듯 자연스럽게 작성하세요.
```
</FileBlock>

<Callout type="tip">
본문이 길어질수록 로드 시 토큰을 많이 써요. 꼭 필요한 지침만 본문에 두고, 세부 참고 자료는 `references/` 폴더로 분리하는 게 좋아요.
</Callout>

<Tabs.Tab>

YAML 프론트매터는 `SKILL.md` 파일 맨 위, `---` 마커 사이에 적는 짧은 설정이에요.\
Claude가 이 내용을 읽고 스킬을 언제, 어떻게 쓸지 판단해요.

### 모든 필드 한눈에 보기

더 세밀하게 동작을 제어하고 싶을 때 아래 필드를 추가할 수 있어요.\
**모든 필드는 선택 사항이에요.** `description`만 권장해요.

<FileBlock filename="~/.claude/skills/full-stack-auditor/SKILL.md">
```yaml
---
name: full-stack-auditor
description: 풀스택 보안 감사를 수행합니다. /audit [repo-path] 로 명시 호출하거나, 보안 관련 대화에서 자동 활성화됩니다.
argument-hint: [repo-path] [severity-level]
disable-model-invocation: false
user-invocable: true
allowed-tools: Read, Grep, Bash, Write
model: claude-opus-4-5
context: fork
agent: coding
hooks:
  pre-tool-call:
    - command: echo "[$(date)] START $TOOL_NAME" >> ~/audit.log
  post-tool-call:
    - command: echo "[$(date)] END $TOOL_NAME" >> ~/audit.log
  stop:
    - command: cat ~/audit.log | mail -s "Audit complete" admin@acme.com
---
```
</FileBlock>

<summary><code>name</code> <small>선택</small></summary>

스킬 표시 이름이에요. 없으면 폴더 이름을 씁니다. 소문자·숫자·하이픈만, 최대 64자.

`claude` 또는 `anthropic`은 예약어로 이름에 사용할 수 없어요.

<details>
<summary><code>description</code> <small>✅ 권장</small></summary>

이 스킬이 무엇을 하고 언제 쓰는지 설명해요. Claude가 이걸 읽고 스스로 로드할지 결정해요 — 즉, **스킬이 트리거되는 기준**이에요. "언제 이 스킬을 써야 하는지"가 명확할수록 Claude가 정확한 타이밍에 꺼내줘요. 없으면 본문 첫 단락을 씁니다.

Anthropic 엔지니어링 블로그에 따르면:

> "이 메타데이터는 Claude가 모든 내용을 컨텍스트에 로드하지 않고도, 각 스킬을 언제 사용해야 하는지 판단할 수 있도록 충분한 정보를 제공합니다."

이는 **점진적 공개(Progressive Disclosure)**의 첫 번째 단계입니다.

##### 구조

```
[What it does] + [When to use it] + [Key capabilities]
```

##### 좋은 예시

```yaml
# Good - 구체적이고 실행 가능함
description: Figma 디자인 파일을 분석하고 개발자용 핸드오프
  문서를 생성합니다. 사용자가 .fig 파일을 업로드하거나
  "디자인 스펙", "컴포넌트 문서", "디자인-코드 핸드오프"를
  요청할 때 사용합니다.

# Good - 트리거 문구 포함
description: 스프린트 계획, 작업 생성, 상태 추적을 포함한
  Linear 프로젝트 워크플로우를 관리합니다. 사용자가 "스프린트",
  "Linear 작업", "프로젝트 계획"을 언급하거나 "티켓 생성"을
  요청할 때 사용합니다.

# Good - 명확한 가치 제안
description: PayFlow를 위한 엔드투엔드 고객 온보딩 워크플로우를
  제공합니다. 계정 생성, 결제 설정, 구독 관리를 처리합니다.
  사용자가 "신규 고객 온보딩", "구독 설정",
  "PayFlow 계정 생성"을 요청할 때 사용합니다.
```

##### 나쁜 예시

```yaml
# Too vague (너무 모호함)
description: 프로젝트를 도와줍니다.

# Missing triggers (트리거 조건 없음)
description: 정교한 다중 페이지 문서 시스템을 생성합니다.

# Too technical, no user triggers (너무 기술적이며 사용자 트리거 없음)
description: 계층적 관계를 갖는 Project 엔티티 모델을 구현합니다.
```

<details>
<summary><code>argument-hint</code> <small>선택</small></summary>

`/스킬명` 입력 시 자동완성 힌트를 표시해요. 아래 예시처럼 쓰면 입력창에 힌트 텍스트가 보여요.

<FileBlock filename="SKILL.md — 이슈 조회 스킬">
```yaml
---
name: issue-detail
description: Linear 이슈 상세 내용을 조회합니다.
argument-hint: "[이슈번호]"
---
```
</FileBlock>

<FileBlock filename="SKILL.md — 파일 변환 스킬">
```yaml
---
name: convert-file
description: 파일을 지정한 형식으로 변환합니다.
argument-hint: "[파일명] [변환형식]"
---
```
</FileBlock>

이렇게 설정하면 Claude Code에서 `/`를 입력할 때 자동완성 드롭다운에 아래처럼 표시돼요.

<SlashBlock>
```
> /issue-detail   [이슈번호]
> /convert-file   [파일명] [변환형식]
```
</SlashBlock>

명령어 이름 오른쪽에 회색으로 힌트 텍스트가 붙어요. 사용자가 어떤 값을 이어서 입력해야 하는지 안내하는 역할이에요.

<Callout type="info">
`argument-hint`는 **표시용 힌트**일 뿐이에요. Claude가 인수를 파싱하거나 강제하지는 않아요.\
사용자가 `/issue-detail ABC-123`처럼 입력하면, 그 전체 문자열이 스킬 본문에 그대로 전달돼요.
</Callout>

</details>

<summary><code>compatibility</code> <small>선택</small></summary>

이 스킬이 실행되기 위해 필요한 환경 조건을 설명해요. 1~500자. 어떤 제품에서 동작하는지, 어떤 시스템 패키지나 네트워크가 필요한지 적으면 돼요.

사용자가 스킬을 설치하기 전에 "내 환경에서 동작할까?"를 판단할 수 있게 해줘요. 특별한 요구사항이 없는 스킬은 생략해도 돼요.

<FileBlock filename="SKILL.md — 플랫폼 제한이 있는 경우">
```yaml
---
name: figma-handoff
description: Figma 디자인을 분석해 개발자용 핸드오프 문서를 생성합니다.
compatibility: "Claude.ai 또는 Claude Code에서만 동작합니다. Figma MCP 서버 연결이 필요해요 (Settings → Extensions → Figma)."
---
```
</FileBlock>

<FileBlock filename="SKILL.md — API 베타 기능이 필요한 경우">
```yaml
---
name: data-pipeline
description: 대용량 CSV 데이터를 분석하고 시각화합니다.
compatibility: "Code Execution Tool 베타가 활성화된 환경에서만 동작합니다. API 사용 시 Anthropic에 베타 접근 권한을 신청하세요."
---
```
</FileBlock>

</details>

<details>
<summary><code>disable-model-invocation</code> <small>선택</small></summary>

`true`로 설정하면 Claude가 자동으로 꺼내지 않아요. `/이름`으로 직접 호출할 때만 실행하고 싶을 때 써요. 기본값: `false`

**언제 필요한가요?**

| 동작 | 호출 주체 | 적합한 상황 | 예시 스킬 | 이유 |
|------|:---------:|-------------|-----------|------|
| 모델(Claude)이 자동 호출 불가 | 사용자만 | 부작용이 있거나(배포·커밋·메시지 전송), 타이밍을 사용자가 통제해야 하는 워크플로우 | `/commit`, `/deploy`, `/send-slack-message` | "코드가 좋아 보인다"는 이유로 Claude가 멋대로 실행하지 못하게 막음 |

</details>

<details>
<summary><code>user-invocable</code> <small>선택</small></summary>

`false`로 설정하면 `/` 메뉴 목록에서 숨겨져요. 사용자가 직접 부르지 않는 배경 지식 스킬에 써요. 기본값: `true`

**언제 필요한가요?**

| 동작 | 호출 주체 | 적합한 상황 | 예시 스킬 | 이유 |
|------|:---------:|-------------|-----------|------|
| 사용자가 명령으로 호출 불가 | Claude만 | 실행 명령이 아니라 배경 지식·컨텍스트로만 쓰는 스킬 | `legacy-system-context` | 사용자는 굳이 실행할 필요 없고, Claude가 관련 상황에서만 자동으로 참고하도록 함 |

</details>

<details>
<summary><code>allowed-tools</code> <small>선택</small></summary>

<Callout type="info">
"허락 요청 없이 쓸 수 있는 도구"인지, "사용 자체를 제한하는 도구"인지 동작 방식 검증이 필요해요.
</Callout>

아래 예시는 파일을 읽을 수는 있지만 수정은 할 수 없는 읽기 전용 모드를 만들어요.

```yaml
---
```
</FileBlock>

<details>
<summary><code>context</code> <small>선택</small></summary>

`fork`로 설정하면 현재 대화와 분리된 독립 컨텍스트에서 스킬을 실행해요. 스킬 콘텐츠가 서브에이전트를 구동하는 프롬프트가 되고, 대화 기록에는 접근할 수 없어요.

<Callout type="warning">
`context: fork`는 실행할 작업이 명시된 스킬에만 의미가 있어요. "이 API 규칙을 사용하세요" 같은 지침만 있고 작업이 없으면, 서브에이전트는 지침을 받지만 실행할 프롬프트가 없어서 의미 있는 출력 없이 반환돼요.
</Callout>

Skills와 서브에이전트는 두 방향으로 함께 작동해요.

**방향 1 — 스킬이 서브에이전트를 실행**

`context: fork`를 설정한 스킬을 호출하면, 새로운 독립 에이전트가 생성돼요.\
스킬 본문이 그 에이전트의 할 일이 되고, `agent` 필드로 어떤 에이전트를 쓸지 고를 수 있어요.

**방향 2 — 서브에이전트가 스킬을 참조**

반대로, 서브에이전트를 만들 때 `skills` 필드로 어떤 스킬을 미리 불러올지 지정할 수 있어요.\
에이전트가 실행될 때 그 스킬들이 배경 지식으로 함께 로드돼요.

이 방향은 [Subagents](/ko/sub-agents#preload-skills-into-subagents) 문서를 참고하세요.

</details>

<details>
<summary><code>agent</code> <small>선택</small></summary>

`context: fork`와 함께 써요. 독립 컨텍스트에서 실행할 서브에이전트 유형을 지정해요. 생략하면 `general-purpose`를 사용해요.

내장 에이전트(`Explore`, `Plan`, `general-purpose`) 또는 `.claude/agents/`에 정의한 커스텀 서브에이전트 이름을 쓸 수 있어요.

**예시 — Explore 에이전트로 코드베이스 리서치 스킬**

<FileBlock filename="~/.claude/skills/deep-research/SKILL.md">
```yaml
---
name: deep-research
description: 주제를 철저히 조사
context: fork
agent: Explore
---

### `$ARGUMENTS`를 철저히 조사하기

1. Glob과 Grep을 사용해 관련 파일을 찾는다
2. 코드를 읽고 분석한다
3. 구체적인 파일 참조(파일 경로/이름 등)를 포함해 조사 결과를 요약한다
```
</FileBlock>

이 스킬이 실행되면:

1. 새로운 격리된 컨텍스트가 생성돼요
2. 서브에이전트는 스킬 본문(`$ARGUMENTS`를 철저히 조사하기…)을 프롬프트로 받아요
3. `agent` 필드가 실행 환경(모델, 도구, 권한)을 결정해요
4. 결과가 요약되어 주 대화로 반환돼요

</details>

<details>
<summary><code>hooks</code> <small>선택</small></summary>

스킬 시작·종료 시 자동으로 실행되는 추가 동작을 정의해요.

</details>


<Callout type="warning">
프론트매터는 Claude의 시스템 프롬프트에 직접 포함돼요. XML 꺾쇠 괄호(`<`, `>`)는 모든 필드에서 사용하지 마세요. 의도치 않은 지침이 삽입될 수 있어요.
</Callout>

## 변수 사용하기

스킬 본문 안에서 특정 변수를 쓰면, 실행 시점에 실제 값으로 자동 교체돼요.

| 변수 | 설명 및 예시 |
|:-----|:-----|
| `$ARGUMENTS` | 스킬을 호출할 때 전달한 모든 인수예요. 본문에 없으면 맨 아래에 `ARGUMENTS: <값>` 형태로 자동 추가돼요. 예: `/review auth.ts` 입력 시 → `auth.ts`로 치환 |
| `$ARGUMENTS[N]` | 특정 인수만 꺼낼 수 있어요. `$ARGUMENTS[0]`은 첫 번째, `$ARGUMENTS[1]`은 두 번째 인수예요. 예: `/convert report.pdf html` 입력 시 → `$ARGUMENTS[0]` = `report.pdf`, `$ARGUMENTS[1]` = `html` |
| `$N` | `$ARGUMENTS[N]`의 축약 표현이에요. `$0` = 첫 번째, `$1` = 두 번째 인수예요. |
| `${CLAUDE_SESSION_ID}` | 현재 대화 세션 ID예요. 예: `sess_a1b2c3d4`. 로그 파일 이름이나 세션별 출력 파일을 만들 때 유용해요. |
| `${CLAUDE_SKILL_DIR}` | 이 스킬의 `SKILL.md`가 들어 있는 폴더 경로예요. 스킬과 함께 묶인 스크립트나 파일을 참조할 때 사용해요. 예: `/Users/john/.claude/skills/session-logger` |

## 파일 연결하기

스킬 폴더에는 `SKILL.md` 외에 추가 파일을 넣을 수 있어요.\
`SKILL.md`는 핵심 지침에만 집중하고, 세부 참고 자료는 별도 파일로 분리해두면 Claude가 필요할 때만 꺼내 읽어요. 매번 전부 불러오지 않아도 돼요.


`SKILL.md` 본문에서 지원 파일을 언급해두면, Claude가 각 파일이 무엇인지, 언제 꺼내야 하는지 알 수 있어요.

<FileBlock filename="SKILL.md">
```markdown
## 추가 자료

- 코드 스타일 규칙은 [references/style-guide.md](references/style-guide.md)를 참고하세요.
- 리뷰 결과 템플릿은 [assets/report.md](assets/report.md)를 참고하세요.
- 검사 스크립트는 [scripts/check.py](scripts/check.py)를 실행하세요.
```
</FileBlock>

<Callout type="tip">
`SKILL.md`는 500줄 이하로 유지하세요. 분량이 많아지면 세부 참고 자료를 별도 파일로 분리하는 게 좋아요.
</Callout>

## 명령어 결과 넣기

스킬 본문에 `` !`명령어` `` 형식으로 쓰면, 스킬이 실행되기 전에 해당 명령어가 먼저 실행돼요.\
명령어 출력 결과가 그 자리에 삽입된 뒤 Claude에게 전달되기 때문에, Claude는 명령어가 아닌 **실제 데이터**를 받아요.

**실행 순서**

1. `` !`명령어` ``가 먼저 실행돼요 — Claude가 내용을 보기 전에
2. 출력 결과가 해당 위치에 채워져요
3. Claude는 데이터가 완성된 최종 내용을 받아요

**어떤 명령어를 쓸 수 있나요?**

운영체제의 셸(macOS·Linux는 bash/zsh, Windows는 PowerShell)에서 실행 가능한 명령어라면 모두 쓸 수 있어요. 단, 입력을 기다리거나 실행 시간이 긴 명령어는 피하세요.

| 카테고리 | 명령어 예시 | 활용 예시 |
|----------|------------|-----------|
| 날짜·시간 | `` !`date "+%Y-%m-%d"` `` | 회의록·보고서에 오늘 날짜 자동 삽입 |
| 현재 폴더 | `` !`pwd` `` | 작업 중인 프로젝트 경로를 컨텍스트로 전달 |
| 파일 목록 | `` !`ls src/` `` | 특정 폴더 안의 파일 목록을 Claude에게 알려주기 |
| 파일 내용 | `` !`cat package.json` `` | 설정 파일 전체를 컨텍스트로 넘기기 |
| Git 히스토리 | `` !`git log --oneline -10` `` | 최근 변경 내역 기반으로 릴리즈 노트 작성 |
| Git 브랜치 | `` !`git branch --show-current` `` | 현재 브랜치명을 작업 컨텍스트로 활용 |
| 환경 변수 | `` !`echo $NODE_ENV` `` | 개발/운영 환경에 따라 다른 지침 적용 |
| 외부 API | `` !`curl -s https://api.example.com/status` `` | 실시간 외부 데이터를 스킬에 주입 |
| 현재 사용자 | `` !`whoami` `` | 사용자 이름을 응답에 포함 |
| 커스텀 스크립트 | `` !`python3 ${CLAUDE_SKILL_DIR}/scripts/check.py` `` | 스킬 폴더의 `scripts/`에 넣어둔 나만의 스크립트 실행 |

**테스트 스크립트 예시**

<FileBlock filename="test-skill.sh">
```bash
#!/bin/bash
# explain-code 스킬 트리거 테스트

PASS=0
FAIL=0

run_test() {
  local description="$1"
  local prompt="$2"
  local should_trigger="$3"

  result=$(claude -p "$prompt" --output-format json 2>/dev/null)
  skill_used=$(echo "$result" | jq -r '.skill // empty')

  if [ "$should_trigger" = "true" ] && [ "$skill_used" = "explain-code" ]; then
    echo "✅ PASS: $description"
    ((PASS++))
  elif [ "$should_trigger" = "false" ] && [ -z "$skill_used" ]; then
    echo "✅ PASS: $description"
    ((PASS++))
  else
    echo "❌ FAIL: $description"
    ((FAIL++))
  fi
}

# 트리거되어야 하는 케이스
run_test "코드 설명 요청" "이 함수가 어떻게 동작해?" true
run_test "동작 원리 질문" "JWT 인증이 뭔지 설명해줘" true

# 트리거되면 안 되는 케이스
run_test "무관한 요청" "오늘 날씨 어때?" false

echo ""
echo "결과: ${PASS}개 통과 / ${FAIL}개 실패"
```
</FileBlock>

```python
client = anthropic.Anthropic()

test_cases = [
    {
        "prompt": "이 코드가 어떻게 동작하는지 설명해줘",
        "expected_skill": "explain-code",
        "checks": ["비유", "ASCII", "단계"]  # 응답에 포함되어야 할 키워드
    },
    {
        "prompt": "오늘 날씨 어때?",
        "expected_skill": None,  # 트리거되면 안 됨
        "checks": []
    }
]

for case in test_cases:
    response = client.messages.create(
        model="claude-sonnet-4-6",
        max_tokens=1024,
        messages=[{"role": "user", "content": case["prompt"]}]
    )
    text = response.content[0].text
    passed = all(kw in text for kw in case["checks"])
    status = "✅" if passed else "❌"
    print(f"{status} {case['prompt'][:30]}...")
```

## <StepBadge step={7} /> 피드백 기반 반복 개선

스킬을 처음 만들었다고 끝난 게 아니에요. 스킬은 **살아있는 문서**예요. 실제로 써보면서 Claude가 어떻게 반응하는지 관찰하고, 신호에 따라 조금씩 다듬어가는 것이 좋은 스킬을 만드는 방법이에요.

<Callout type="info">
"스킬이 완벽해야 배포한다"는 생각은 버리세요. 70%만 완성해도 써보고, 실제 사용에서 나온 신호로 나머지 30%를 채우는 게 훨씬 효과적이에요.
</Callout>

개선이 필요한 신호는 크게 세 가지예요.

<details>
<summary>**신호 1 — 스킬이 실행되어야 할 때 실행되지 않음 (언더 트리거링)**</summary>

이런 일이 생기면 언더 트리거링을 의심하세요.

- Claude가 스킬을 자동으로 꺼내지 않아서, 사용자가 매번 `/스킬명`으로 직접 불러야 해요
- "이 스킬 언제 써요?"라는 질문이 생겨요
- 트리거 문구를 바꿔가며 여러 번 시도해야 스킬이 실행돼요

**왜 이런 일이 생기나요?**

Claude는 `description` 필드를 읽고 "지금 이 스킬을 꺼내야 할 타이밍인가?"를 판단해요. description이 너무 짧거나 모호하면, 사용자가 말하는 방식과 description의 키워드가 맞지 않아 Claude가 스킬을 인식하지 못해요.

**해결 방법**

`description`에 사용자가 실제로 쓰는 표현을 더 넣어요. "언제 이 스킬을 써야 하는지"가 구체적일수록 Claude가 정확한 타이밍에 꺼내줘요.

<FileBlock filename="SKILL.md — 수정 전">
```yaml
# ❌ 너무 짧고 모호함
description: 코드를 설명합니다.
```
</FileBlock>

<FileBlock filename="SKILL.md — 수정 후">
```yaml
# ✅ 구체적인 트리거 문구 포함
description: 시각적 다이어그램과 비유로 코드를 설명합니다.
  "이 코드 어떻게 동작해?", "이거 설명해줘", "이 함수가 뭐 하는 거야?" 같은
  질문이 나올 때 사용합니다.
```
</FileBlock>

</details>

<details>
<summary>**신호 2 — 관련 없는 상황에도 스킬이 실행됨 (오버 트리거링)**</summary>

이런 일이 생기면 오버 트리거링을 의심하세요.

- 전혀 관계없는 요청인데도 스킬이 실행돼서 엉뚱한 응답이 나와요
- 사용자가 스킬을 끄거나 삭제해버려요
- "이 스킬 왜 이렇게 자꾸 나와요?"라는 반응이 나와요

**왜 이런 일이 생기나요?**

description에 너무 일반적인 키워드가 들어 있으면, 그 키워드가 조금이라도 포함된 대화에서는 전부 스킬이 실행돼요. 예를 들어 description에 그냥 "코드"라고만 써놓으면, 코드 관련 모든 대화에서 스킬이 작동할 수 있어요.

**해결 방법**

description을 더 구체적으로 좁히거나, 스킬을 사용하면 안 되는 상황을 명시해요. 또는 Claude가 자동으로 트리거하지 못하도록 `disable-model-invocation: true`를 설정하고 `/스킬명`으로만 호출하게 해요.

<FileBlock filename="SKILL.md — 수정 전">
```yaml
# ❌ 너무 넓은 범위
description: 코드를 분석하고 개선합니다.
```
</FileBlock>

<FileBlock filename="SKILL.md — 수정 후">
```yaml
# ✅ 명확한 범위 + 제외 조건
description: 코드 보안 취약점을 분석합니다.
  사용자가 "보안 감사", "취약점 검사", "security audit"을 요청할 때만 사용합니다.
  일반 코드 리뷰, 버그 수정, 기능 구현 요청에는 사용하지 않습니다.
```
</FileBlock>

</details>

<details>
<summary>**신호 3 — 실행은 되지만 결과가 일관되지 않음 (실행 문제)**</summary>

이런 일이 생기면 실행 품질 문제를 의심하세요.

- 같은 요청인데 어떨 때는 잘 되고 어떨 때는 이상한 결과가 나와요
- API나 외부 서비스 호출이 실패해요
- 사용자가 스킬 응답을 받고 나서 수정 요청을 자주 해요

**왜 이런 일이 생기나요?**

SKILL.md의 지침이 너무 추상적이거나 단계 사이에 빠진 내용이 있으면, Claude가 판단해서 채워 넣는데 그 판단이 매번 다를 수 있어요. 외부 서비스를 쓰는 스킬이라면 오류 상황에 대한 안내가 없어서 실패할 때 어떻게 해야 할지 모르는 경우도 있어요.

**해결 방법**

지침의 각 단계를 더 구체적으로 작성하고, 예상되는 오류 상황과 대처 방법을 추가해요.

<FileBlock filename="SKILL.md — 수정 전">
```markdown
## Instructions

1. 데이터를 가져온다
2. 분석한다
3. 결과를 보고한다
```
</FileBlock>

<FileBlock filename="SKILL.md — 수정 후">
```markdown
## Instructions

### 1단계: 데이터 가져오기
MCP를 통해 Linear에서 현재 스프린트 이슈 목록을 가져온다.
- 실패 시: "Linear 연결을 확인해줘"라고 안내하고 중단한다.

### 2단계: 분석
이슈를 우선순위(P0, P1, P2)와 예상 시간 기준으로 분류한다.
우선순위 표시가 없는 이슈는 P2로 간주한다.

### 3단계: 보고
마크다운 표 형식으로 분류 결과를 출력한다.
각 이슈에는 Linear 링크를 포함한다.
```
</FileBlock>

</details>

세 신호 중 가장 먼저 잡아야 할 건 **언더 트리거링**이에요. 스킬이 실행조차 안 되면 다른 개선은 의미가 없어요. description을 다듬는 것부터 시작하세요.

## <StepBadge step={8} /> 배포 및 공유

스킬은 만들고 나서 혼자만 쓰는 것보다, 공유했을 때 진정한 가치가 생겨요. 잘 만든 스킬은 팀 전체의 생산성을 높이고, MCP 서비스를 제공한다면 스킬이 포함된 경우 사용자가 더 빠르게 가치를 경험할 수 있어서 경쟁 우위가 돼요.

| 방법 | 할 일 | 주의사항 |
|------|-------|----------|

## 스킬이 로드되는 3단계

스킬은 **3단계 구조**로 작동해요. 필요한 정보만 그때그때 로드해서 토큰 사용량을 최소화해요.

**1단계 — YAML 프론트매터**

항상 Claude의 시스템 프롬프트에 로드돼요.\
각 스킬을 언제 써야 하는지 판단할 수 있도록 최소한의 정보만 담아요. 전체 내용은 아직 불러오지 않아요.

**2단계 — SKILL.md 본문**

Claude가 현재 작업과 관련 있다고 판단할 때 로드돼요.\
전체 지침과 상세 가이드를 담아요.

**3단계 — 연결된 파일들**

스킬 디렉토리에 포함된 추가 파일(`scripts/`, `references/`, `assets/`)로, 필요할 때만 선택적으로 탐색·로드돼요.

<Callout type="info">
점진적 공개 방식 덕분에 스킬이 많아도 토큰 낭비 없이 전문적인 기능을 유지할 수 있어요.
</Callout>
