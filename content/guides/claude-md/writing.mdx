# CLAUDE.md 작성법

## 이런 상상을 해봅시다

당신에게 아주 유능한 팀원이 생겼습니다.
모르는 게 없고, 시키는 건 뭐든 빠르게 해결해줘요.

그런데 한 가지 문제가 있습니다. **이 팀원은 매일 아침, 어제의 기억을 모두 잃고 출근합니다.**

첫날, 당신은 꼼꼼하게 설명했어요.

> "우리 서비스 이름은 ○○이고, 고객은 이런 분들이고, 문서는 이 형식으로 써야 해."

팀원은 완벽하게 이해하고 일을 해냈습니다.

다음 날 아침, 팀원이 다시 왔습니다.

> "안녕하세요! 저 오늘 처음 왔는데요, 뭘 도와드릴까요?"

어제 했던 설명을 또 해야 합니다. 그다음 날도, 그다음 날도.

**CLAUDE.md는 이 문제를 해결합니다.**
팀원 책상 위에 놓인 업무 안내서처럼, Claude가 매일 가장 먼저 읽는 파일입니다.
프로젝트의 규칙, 작업 방식, 중요한 맥락을 한 번만 적어두면
Claude는 매 세션마다 그 내용을 바탕으로 대화를 시작합니다.

---

## Claude가 프로젝트를 기억하는 방법

Claude Code 세션에 처음 진입할 때마다 새로운 컨텍스트 윈도우로 시작됩니다. 즉, Claude는 이전 대화 내용을 기억하지 못한 채 매번 새로 시작합니다.<br/>
**CLAUDE.md 파일**은 세션이 시작될 때마다 자동으로 Claude에 전달되어 지속적인 지침을 제공합니다.

---

## CLAUDE.md란

CLAUDE.md 파일은 프로젝트, 개인 워크플로우 또는 전체 조직에 대해 Claude에 지속적인 지침을 제공하는 마크다운 파일입니다.
이러한 파일을 일반 텍스트로 작성하면 Claude가 모든 세션의 시작 시 이를 읽습니다.

Claude는 이 내용을 절대적인 명령이 아닌 참고 자료로 활용합니다 — 지침이 모호하거나 서로 충돌하면 Claude가 스스로 판단해 처리할 수 있어요.<br/>
지침이 더 구체적이고 간결할수록 Claude가 더 일관되게 따릅니다.

| | CLAUDE.md 파일 |
| :--- | :--- |
| **작성자** | 사용자 |
| **포함 내용** | 지침 및 규칙 |
| **범위** | 프로젝트, 사용자 또는 조직 |
| **로드 대상** | 모든 세션 |
| **사용 목적** | 코딩 표준, 워크플로우, 프로젝트 아키텍처 |

Claude의 동작을 안내하려면 CLAUDE.md 파일을 사용합니다.

---

## CLAUDE.md 파일을 어디에 배치할지 선택

CLAUDE.md 파일은 여러 위치에 있을 수 있으며, 각 위치는 다른 범위를 가집니다.<br/>
더 구체적인 위치가 더 광범위한 위치보다 우선합니다.

| 범위 | 위치 | 공유 대상 |
| :--- | :--- | :--- |
| **프로젝트 지침** | `./CLAUDE.md` 또는 `./.claude/CLAUDE.md` | 소스 제어를 통한 팀 멤버 |
| **사용자 지침** | `~/.claude/CLAUDE.md` | 본인만(모든 프로젝트) |
| **로컬 지침** | `./CLAUDE.local.md` | 본인만(현재 프로젝트) |

처음 시작한다면 **프로젝트 지침**(`./CLAUDE.md`)부터 만들어보세요. 프로젝트 루트에 파일 하나만 만들면 바로 사용할 수 있고, 팀원과도 공유할 수 있어 가장 효과적인 출발점입니다.

범위 개념이 낯설다면 [설정 범위(Scope)](/guides/environment-setup/scope)를 함께 참고하세요.

## CLAUDE.md 파일 생성

처음부터 직접 작성하지 않아도 됩니다. `/init` 명령어를 실행하면 Claude가 코드베이스를 분석해 CLAUDE.md를 자동으로 만들어줍니다.

<SlashBlock>
```
> /init
```
</SlashBlock>

자세한 사용법은 [/init 레퍼런스](/reference/commands/init)를 참고하세요.

Claude가 프로젝트를 살펴보고 빌드 명령, 테스트 지침, 프로젝트 규칙 등을 담은 파일을 생성합니다.

코드가 없는 빈 프로젝트에서 실행하면 Claude가 먼저 확인을 요청해요.

<ClaudeBlock>
```
⏺ The directory appears to be empty. There's no code to analyze, so there's
  nothing to document in a CLAUDE.md.

  Would you like me to initialize a minimal CLAUDE.md as a placeholder for when
  you add code to this project?
```
</ClaudeBlock>

`yes`를 입력하면 최소한의 CLAUDE.md가 만들어져요.

<VibeBlock>
```
> yes
```
</VibeBlock>

<FileBlock filename="CLAUDE.md">
```markdown
# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.
```
</FileBlock>

이 파일을 출발점 삼아 직접 내용을 채워나가면 됩니다.

CLAUDE.md가 이미 존재하면 `/init`은 파일을 덮어쓰지 않고 개선할 수 있는 사항을 제안합니다. Claude가 스스로 발견하지 못한 지침이 있다면 그 제안을 바탕으로 내용을 보완해나가세요.

> 실수를 발견할 때마다 CLAUDE.md에 한 줄 추가.<br/>처음부터 완벽하게 쓰려고 하지 마세요.
>
> **— Boris Cherny (클로드코드 초기 개발자)**

## 효과적인 지침 작성

CLAUDE.md 파일은 **매 세션이 시작될 때마다 Claude에게 전달되어 대화 공간을 차지**합니다.<br/>
Claude는 이 파일을 강제 명령이 아닌 참고 자료로 읽기 때문에, **지침을 작성하는 방식**이 Claude가 얼마나 잘 따르는지에 직접 영향을 미칩니다.<br/>
구체적이고 간결하며 잘 구조화된 지침이 가장 효과적입니다.

### 구조

마크다운 헤더와 글머리 기호를 사용해 관련 지침을 그룹화합니다.<br/>
Claude도 사람처럼 문서 구조를 훑어봅니다. 잘 정리된 섹션이 빽빽한 단락보다 따르기 쉽습니다.

<FileBlock filename="CLAUDE.md">
```markdown
## 코딩 스타일

- API 호출: src/api/

## 커밋 규칙
- 커밋 전에 반드시 `npm test` 실행
- 커밋 메시지는 한국어로 작성
```
</FileBlock>

### 도메인 용어 정의

Claude Code는 일반적인 프로그래밍 원칙은 잘 이해하지만, 프로젝트 고유의 비즈니스 용어는 모를 수 있습니다.<br/>
특히 다음과 같은 경우 Claude가 혼란스러울 수 있습니다.

- 프로젝트 고유의 전문 용어나 엔티티 이름
- 일반적인 단어와 다른 의미로 쓰이는 표현
- 맥락 없이는 뜻을 알 수 없는 약어

CLAUDE.md에 도메인 용어를 정의해두면 Claude가 코드베이스를 더 정확하게 탐색하고 올바른 파일을 편집할 수 있습니다.

**피트니스 앱 예시**

<FileBlock filename="CLAUDE.md">
```markdown
## Terminology
- **Wraps:** 사용자를 위한 연간 비디오 요약의 다른 이름
- **Athlete:** 시스템에서 Users로도 불림
- **Track:** GPX 파일에서 러너의 경로를 구성하는 세그먼트 모음
- **Pace:** 달리기 속도. 항상 `min/km` 또는 `min/mile` 단위 사용
```
</FileBlock>

**데이터 파이프라인 예시**

<FileBlock filename="CLAUDE.md">
```markdown
## Terminology
- "Module"은 src/modules의 파이프라인을 의미함. 일반적인 JS 모듈이 아님.
```
</FileBlock>

이렇게 정의해두면 프롬프트에서 도메인 용어를 바로 사용할 수 있습니다.<br/>
예를 들어 "Wraps 페이지 수정해줘"라고만 해도 Claude가 연간 비디오 요약 기능임을 알고 올바른 파일을 찾아갑니다.

핵심은 **용어가 코드의 어디에 매핑되는지**까지 함께 적어주는 것입니다.<br/>
용어의 의미뿐 아니라 실제 파일 위치를 알아야 Claude가 올바르게 작업할 수 있습니다.

<FileBlock filename="CLAUDE.md — 실제 적용 예시">
```markdown
## Terminology
- **청구서(Invoice):** 월별 정산 문서. Bill과 혼용하지 말 것
- **파트너:** 외부 제휴 업체. 코드에서는 `Partner` 모델 (src/models/partner.ts)
- **정산:** 파트너에게 수수료를 지급하는 프로세스. Settlement 모듈 참조
- **PG:** Payment Gateway의 약어. src/services/pg/ 디렉토리
```
</FileBlock>

## `@` 파일 참조

CLAUDE.md 파일은 `@path/to/import` 구문을 사용해 다른 파일을 가져올 수 있습니다.<br/>
가져온 파일은 CLAUDE.md와 함께 세션 시작 시 컨텍스트에 함께 로드됩니다.

상대 경로와 절대 경로 모두 허용되며, 상대 경로는 작업 디렉토리가 아닌 **가져오기를 포함하는 파일 기준**으로 해석됩니다.<br/>
가져온 파일은 최대 5단계 깊이까지 다른 파일을 재귀적으로 가져올 수 있습니다.

<FileBlock filename="CLAUDE.md">
```markdown
프로젝트 개요는 @README를 참조하고
사용 가능한 npm 명령은 @package.json을 참조합니다.

# 추가 지침
- git 워크플로우: @docs/git-instructions.md
```
</FileBlock>

버전 관리에 올리지 않을 개인적인 설정은 `CLAUDE.local.md`를 사용합니다. 자동으로 로드되고 `.gitignore`에 추가됩니다.

git worktree는 브랜치마다 **완전히 새로운 폴더**를 만들어 동시에 열어두는 방식입니다.<br/>
<span style={{background:'#fef08a',padding:'1px 4px',borderRadius:'3px'}}>`CLAUDE.local.md`는 `.gitignore`에 등록된 파일이기 때문에 새 폴더에 따라오지 않습니다.</span>

worktree로 새 폴더를 만들면 git이 해당 브랜치의 파일들을 가져오는데,
`.gitignore`에 있는 파일은 git이 추적하지 않으니 새 폴더에 복사되지 않습니다.

| | git이 관리하는 파일 | `CLAUDE.local.md` |
| :--- | :--- | :--- |
| **새 worktree 폴더** | ✅ 있음 | ❌ 없음 (git 밖의 파일) |

</FileBlock>

<span style={{background:'#fef08a',padding:'1px 4px',borderRadius:'3px'}}>Claude Code가 프로젝트에서 외부 파일 참조를 처음 만나면 승인 대화를 표시합니다.</span><br/>
거부하면 해당 가져오기는 비활성화된 채로 유지되고 다시 묻지 않습니다.

### `@` 참조 파일은 언제 로드되나요?

**세션 시작 시 즉시, 전부 로드됩니다.** 지연 로딩(lazy loading)이 아닙니다.

<span style={{background:'#fef08a',padding:'1px 4px',borderRadius:'3px'}}>`@` 참조는 현재 작업과의 관련성에 관계없이 대화 시작 시 참조된 모든 파일을 컨텍스트에 로드합니다.</span><br/>
예를 들어 `@docs/authentication.md`를 적어두면, 인증과 전혀 관련 없는 작업을 하더라도 매번 그 파일 전체가 컨텍스트에 들어갑니다.

<Callout type="info">
6개의 참조 파일(약 2,100줄)을 가진 모듈식 구조는 하나의 거대한 파일과 동일한 토큰을 소비합니다.<br/>
`@path` 가져오기는 **파일 관리의 편의성**을 위한 것이지, 토큰 절약을 위한 것이 아닙니다.
</Callout>

<span style={{background:'#fef08a',padding:'1px 4px',borderRadius:'3px'}}>토큰을 절약하면서 필요할 때만 로드하고 싶다면, `@path` 대신 **포인터 방식**을 사용합니다.</span>

<FileBlock filename="CLAUDE.md">
```markdown
## 인증
인증 흐름 상세 내용은 docs/authentication.md 참조.
필요시 해당 파일을 직접 읽을 것.
```
</FileBlock>

이렇게 하면 Claude는 인증 관련 작업이 생겼을 때 스스로 파일을 읽어서 컨텍스트에 가져옵니다.<br/>
`@docs/authentication.md`로 직접 가져오면 매 세션마다 무조건 로드되지만, 포인터만 남겨두면 필요할 때만 읽어 컨텍스트를 절약할 수 있습니다.

## CLAUDE.md 파일에 텍스트 추가하기

프롬프트 앞에 `#`을 붙이면 대화로 처리되지 않고 CLAUDE.md에 바로 추가됩니다.

```
> # API 핸들러는 항상 try-catch로 감쌀 것
```
</VibeBlock>

작업하다가 새로운 규칙이 생각났을 때 파일을 열어 직접 편집하지 않아도 됩니다.<br/>
대화창에 `#`으로 시작하는 메시지를 입력하면 Claude가 해당 내용을 CLAUDE.md에 추가합니다.
