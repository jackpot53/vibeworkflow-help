# CLAUDE.md 베스트 프랙티스

실전에서 검증된 CLAUDE.md 작성 팁들이에요. 처음 접하는 분도 바로 적용할 수 있도록 최대한 쉽게 설명했어요.

## 처음부터 완벽하게 쓰려고 하지 마세요

> *"실수를 발견할 때마다 CLAUDE.md에 한 줄 추가하세요. 처음부터 완벽하게 쓰려고 하지 마세요."*
>
> — Boris Cherny (Claude Code 초기 개발자)

CLAUDE.md는 처음 쓸 때 완성품이 아니어도 돼요. <mark>Claude가 실수할 때마다 한 줄씩 추가</mark>하는 방식으로 자연스럽게 키워나가는 게 오히려 더 좋아요.

예를 들면 이런 식이에요.

- Claude가 커밋 메시지를 영어로 쓴다 → `"커밋 메시지는 한국어로 작성"` 추가
- Claude가 `console.log`를 쓴다 → `"로그는 logger 모듈만 사용"` 추가
- Claude가 잘못된 폴더에 파일을 만든다 → 올바른 경로 추가

억지로 처음부터 완성하려 하면 오히려 쓸모없는 내용이 쌓여요.

## 프로젝트 전문 용어를 정의하세요

Claude는 일반적인 프로그래밍 지식은 풍부하지만, 여러분 프로젝트만의 용어는 모를 수 있어요.

<Callout type="tip">
  용어를 CLAUDE.md에 정의해두면 "Wraps 페이지 수정해줘"라고만 해도 Claude가 어떤 기능인지, 어느 파일인지 바로 찾아가요.
</Callout>

**피트니스 앱 예시:**

<FileBlock filename="CLAUDE.md — 용어 정의">
```markdown
## Terminology
- **Wraps:** 사용자를 위한 연간 비디오 요약. 코드에서는 `YearlyWrap` 모델
- **Athlete:** 시스템에서 `User`로도 불림
- **Track:** GPX 파일에서 러너의 경로를 구성하는 세그먼트 모음
- **Pace:** 달리기 속도. 항상 `min/km` 또는 `min/mile` 단위 사용
```
</FileBlock>

**데이터 파이프라인 예시:**

<FileBlock filename="CLAUDE.md — 용어 정의">
```markdown
## Terminology
- "Module"은 src/modules의 파이프라인을 의미함. 일반적인 JS 모듈이 아님
- "PG": Payment Gateway 약어. src/services/pg/ 디렉토리
- "정산": 파트너에게 수수료를 지급하는 프로세스. Settlement 모듈 참조
```
</FileBlock>

핵심은 **용어가 코드 어디에 매핑되는지**까지 함께 적어주는 거예요. 그래야 Claude가 올바른 파일을 찾아서 작업할 수 있어요.

## CLAUDE.md는 200줄 이하를 목표로 하세요

CLAUDE.md가 길어질수록 Claude가 내용을 **무시할 가능성**이 높아져요.

<Callout type="warning">
  프론티어 LLM은 약 150~200개의 지침을 합리적인 일관성으로 따를 수 있어요. 지침이 많아질수록 준수 품질이 균일하게 낮아집니다.
</Callout>

실제 개발자 사례를 보면:

- Boris Cherny (Claude Code 개발자) → **200줄 미만** 권장
- Dex (humanlayer 개발자) → 더 엄격하게 **60줄**로 줄였는데, 60줄이어도 가끔 무시함 😅

결론: **짧을수록 좋아요.** 지울 수 있는 건 과감히 지우세요.

특히 이런 것들은 Claude에게 시키지 말고 도구에 맡기세요.

```markdown
# ❌ 이렇게 하지 마세요 (린터 역할을 Claude에게 시키는 것)
## Code Style
- 들여쓰기는 2칸
- 세미콜론 항상 사용
- 싱글 쿼트 사용
- 줄 끝 쉼표 사용
```

<Callout type="tip">
  **ESLint**와 **Prettier**는 코드를 자동으로 검사하고 정리해주는 도구예요. 규칙을 한 번 설정해두면 파일을 저장할 때마다 자동으로 적용돼요.

  **Prettier — 코드 형식 자동 정리**

  저장 전 (들쭉날쭉한 형식):
  ```js
  const greet = (name) => {
    console.log("hello "+name)
      return name
  }
  ```

  저장 후 (Prettier가 자동 정리):
  ```js
  const greet = (name) => {
    console.log("hello " + name);
    return name;
  };
  ```

  들여쓰기, 세미콜론, 따옴표, 줄 끝 쉼표 같은 것들을 자동으로 맞춰줘요.

  ---

  **ESLint — 코드 오류 자동 감지**

  작성한 코드:
  ```js
  const userName = "홍길동";
  console.log(userNane); // 오타
  ```

  ESLint 경고:
  ```
  ⚠ 'userNane' is not defined   (오타로 존재하지 않는 변수)
  ⚠ 'userName' is assigned but never used
  ```

  실수로 오타를 낸 변수명, 선언만 하고 쓰지 않는 변수 같은 것들을 실행 전에 잡아줘요.
</Callout>

ESLint, Prettier처럼 **항상 같은 결과를 내주는 전용 도구**가 이미 있는데 굳이 Claude에게 시킬 필요 없어요. Claude는 느리고 비싸요.

## 지침이 커지면 `.claude/rules/`로 분리하세요

CLAUDE.md에 규칙이 계속 쌓이면 Claude가 규칙을 무시하기 시작해요. 실제로 한 개발자는 모노레포 CLAUDE.md가 **47,000단어**까지 불어나 Claude가 느려지고 규칙도 안 따르는 문제를 겪었어요.

해결책은 `.claude/rules/` 디렉토리에 파일을 분리하고, **frontmatter의 `paths` 필드**로 필요한 규칙만 필요한 시점에 로드되게 하는 거예요.

<FileBlock filename=".claude/rules/api-rules.md">
```markdown
---
paths:
  - "src/api/**/*.ts"
---

# API 개발 규칙

- 모든 API 엔드포인트는 입력 검증을 포함해야 합니다
- 표준 오류 응답 형식을 사용합니다
- OpenAPI 문서 주석을 포함합니다
```
</FileBlock>

`paths`를 지정하면 Claude가 `src/api/` 아래 파일을 열 때만 이 규칙이 로드돼요. 관련 없는 작업에서는 컨텍스트를 차지하지 않아요.

여러 패턴을 조합할 수도 있어요.

```markdown
---
paths:
  - "src/**/*.{ts,tsx}"
  - "tests/**/*.test.ts"
---
```

<Callout type="tip">
  `paths` 필드가 없는 규칙은 CLAUDE.md와 마찬가지로 세션 시작 시 무조건 로드돼요.
</Callout>

## 모노레포에는 여러 CLAUDE.md를 사용하세요

모노레포처럼 프로젝트가 여러 패키지로 나뉜 경우, CLAUDE.md도 각 폴더에 맞게 나눠두는 게 좋아요.

Claude Code는 CLAUDE.md를 두 가지 방식으로 로드해요.

**위쪽으로 탐색 (즉시 로드):** 현재 작업 폴더에서 위로 올라가며 경로상의 모든 CLAUDE.md를 찾아 세션 시작 시 즉시 로드해요.

**아래쪽으로 탐색 (지연 로드):** 하위 폴더의 CLAUDE.md는 세션 시작 시 로드하지 않아요. Claude가 그 폴더의 파일을 실제로 열 때만 로드해요.

```
/my-monorepo/
├── CLAUDE.md           ← 공통 규칙 (즉시 로드)
├── frontend/
│   └── CLAUDE.md       ← 프론트엔드 규칙 (지연 로드)
└── backend/
    └── CLAUDE.md       ← 백엔드 규칙 (지연 로드)
```

이 구조에서 백엔드 파일을 열면 `backend/CLAUDE.md`가 그때 로드돼요. 프론트엔드 작업 중엔 백엔드 규칙이 컨텍스트를 차지하지 않아요.

CLAUDE.md가 길어지면 Claude는 앞부분에만 집중하고 뒷부분을 흘려보내는 경향이 있어요. 특정 상황에서만 중요한 규칙은 `<important if="...">` 태그로 감싸면 해당 상황일 때만 Claude가 집중해요.

## 코드 방식을 바꾸기 시작했으면 끝까지 바꾸세요

작업 방식을 바꾸다가 절반만 바꾼 상태로 두면 Claude가 헷갈려요.

예를 들어 사무실 서류 정리 방식을 바꾼다고 해볼게요.

> "앞으로 서류는 파란 폴더에 넣자"고 결정했는데,
> 책상 위엔 파란 폴더 2개, 빨간 폴더 2개가 섞여 있는 상황.

이 상태에서 새 직원이 서류를 가져오면 어떻게 해야 할지 몰라요. "파란 게 맞는 건지, 빨간 게 맞는 건지" 기존 것들을 보고 판단하려 하죠.

Claude도 마찬가지예요. 코드베이스에 **옛날 방식과 새 방식이 섞여 있으면** 어느 쪽이 정답인지 몰라 옛날 방식을 따라갈 수 있어요.

예를 들어 React라는 도구에서 2019년을 기점으로 버튼 같은 UI 요소를 만드는 방식이 바뀌었어요. 둘 다 결과는 같지만 코드 생김새가 달라요.

- **옛날 방식 (Class):** 틀을 만들고 그 안에 규칙들을 채워 넣는 형태. 코드가 길고 복잡해요.
- **새 방식 (Function):** 필요한 것만 바로 적는 형태. 훨씬 짧고 읽기 쉬워요.

<div className="not-prose grid grid-cols-1 sm:grid-cols-2 gap-px bg-border my-6 border border-border text-sm">
  <div className="bg-background px-5 py-4">
    <div className="font-semibold text-red-500 mb-3">❌ Dashboard — 옛날 방식 (Class)</div>
    <pre className="text-xs text-muted-foreground leading-relaxed whitespace-pre">{`class Button extends Component {
  render() {
    return (
      <button
        onClick={this.props.onClick}
      >
        {this.props.label}
      </button>
    )
  }
}`}</pre>
  </div>
  <div className="bg-background px-5 py-4">
    <div className="font-semibold text-emerald-500 mb-3">✅ UserProfile — 새 방식 (Function)</div>
    <pre className="text-xs text-muted-foreground leading-relaxed whitespace-pre">{`function Button({ label, onClick }) {
  return (
    <button onClick={onClick}>
      {label}
    </button>
  )
}`}</pre>
  </div>
</div>

새 방식이 더 짧고 깔끔하지만, 두 방식이 프로젝트 안에 섞여 있으면 Claude는 "어느 쪽이 이 팀의 방식이지?"를 판단할 수 없어요. 기존 파일 절반이 옛날 방식이니 그걸 따라 만들 수도 있어요.

이 상태에서 "새 화면 만들어줘"라고 하면 Claude는 옛날 방식을 골라 만들 수도 있어요.

<mark>방식을 바꾸기 시작했으면 끝까지 바꾸세요.</mark> 코드베이스가 일관될수록 Claude의 결과물 품질이 높아져요.

## "부탁"할 건 CLAUDE.md, "강제"할 건 settings.json

반드시 지켜져야 하는 동작은 CLAUDE.md에 쓰지 말고 `settings.json`으로 강제하세요.

| | CLAUDE.md | settings.json |
| :--- | :--- | :--- |
| **Claude가 가끔 무시할 수 있음?** | 네 | 아니요 |
| **용도** | 가이드라인, 권장 사항 | 권한, 필수 동작 강제 |

**예시: 커밋에 `Co-Authored-By` 제거하기**

```markdown
# ❌ CLAUDE.md에 이렇게 쓰면 가끔 무시됨
NEVER add Co-Authored-By to commit messages
```

```json
// ✅ settings.json으로 강제하면 확실
{
  "attribution": {
    "commit": ""
  }
}
```
